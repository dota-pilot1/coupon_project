========================================
쿠폰 관리 전체 프로세스 개요
========================================

■ 프로세스 흐름 (4단계)

  저장(생성) → 승인 → 발급 → 사용

  1) 저장(생성) : 담당자가 쿠폰 마스터 등록
     - 쿠폰명, 할인유형(정률/정액), 할인값
     - 사용 대상 제한 (점포 → 사이트 → 코너 → 메뉴)
     - 유효 기간 제어 (일자/시간대/요일)
     - 저장 시 승인상태 = C(생성)

  2) 승인 : 상위 결재자가 검토 후 상태 변경
     - C(생성) → W(승인요청) → Y(승인) or R(반려)
     - Y(승인) → T(강제중지) 가능
     - R(반려) → W(재승인요청) 가능
     - Maker-Checker 패턴 (등록자 ≠ 승인자)

  3) 발급 : 승인(Y) 완료된 쿠폰만 발급 가능
     - 발급 수량(N건) 지정 → 개별 쿠폰번호 일괄 생성
     - 발급 마스터(coupon_issuance) + 개별 쿠폰(coupon_issued) INSERT
     - 개별 쿠폰 초기 상태 = UNUSED(미사용)

  4) 사용 : 매장에서 쿠폰 사용 처리
     - UNUSED → USED 상태 변경
     - 사용일시, 사용점포, 사용금액 기록
     - 유효기간 초과 시 EXPIRED(만료) 처리


■ 승인이 발급보다 먼저인 이유

  - 쿠폰은 금전적 가치가 있는 항목 (할인 = 매출 감소)
  - 무분별한 발급 방지를 위해 상위 결재자 검증 필수
  - 기업 시스템의 Maker-Checker 원칙 적용


■ 연습 프로젝트 메뉴 구성

  | 메뉴         | 경로          | 역할                        |
  |-------------|--------------|----------------------------|
  | 쿠폰 마스터  | /            | 1단계 - 쿠폰 등록/수정       |
  | 공통코드     | /common-code | 보조 - 코드성 데이터 관리     |
  | 승인 관리    | /approval    | 2단계 - 승인/반려/중지        |
  | 쿠폰 발급    | /issuance    | 3단계 - 쿠폰번호 일괄 생성    |
  | 사용 현황    | /usage       | 4단계 - 통계/이력 조회        |


■ DB 테이블 관계

  coupon_master (쿠폰 정의)
    ├── cond_shop/site/corner/menu (사용 제한 조건)
    ├── cond_time (시간대/요일 조건)
    ├── coupon_issuance (발급 이력)
    │     └── coupon_issued (개별 쿠폰)
    └── apprvCd 상태: C → W → Y/R → T
